**Date:** 2026-05-08
**Mode:** Quick (Phase 1: Reconnaissance + Phase 4: Engineering Analysis)
**Auditor:** Claude Sonnet 4.6

---

## 1. Project Overview

**What this is:** An internal screenplay analysis dashboard for Lemon Studios. Producers upload screenplay PDFs, an AI pipeline (Anthropic Claude via Firebase Cloud Function) analyzes them, and the results are stored in Firestore. The dashboard provides filtering, scoring, comparison, analytics charts, PDF coverage export, and shareable partner links. The tool is designed to triage 500+ screenplays for producer review.

**Version:** 6.8.16
**Source files:** 215 TypeScript/TSX files, ~14,000 lines
**Test files:** 37 unit test files (463 passing, 8 pre-existing failures), 4 Playwright e2e specs
**Routes:** `/` (dashboard), `/settings`, `/share/:token`
**Deployment:** Firebase Hosting + 2 Cloud Functions (`analyzeScreenplay`, `llmProxy`)

## 2. Architecture Summary

### Stack
| Layer | Technology |
|---|---|
| Frontend | React 19, TypeScript strict, Vite 7 |
| Styling | Tailwind CSS 4 (via @tailwindcss/vite) |
| Client state | Zustand 5 (16 stores) |
| Server state | TanStack React Query 5 (React Query) |
| Routing | React Router 7 |
| Database | Firebase Firestore 12 |
| Auth | Firebase Anonymous Auth |
| Storage | Firebase Storage |
| AI (analysis) | Anthropic Claude via `analyzeScreenplay` Cloud Function |
| AI (chat/posters) | Google Gemini (DevExec overlay, poster generation) |
| LLM proxy | `llmProxy` Cloud Function → Anthropic SDK |
| Charts | Recharts 3 (lazy-loaded) |
| PDF render | @react-pdf/renderer (lazy-chunked) |
| PDF parse | pdfjs-dist |

### Data Flow
```
User uploads PDF
  → pdfParser (pdfjs-dist, browser)
  → analysisService → multiPassAnalysis (V7) or promptClient (V6)
  → proxyClient → llmProxy Cloud Function → Anthropic API
  → saveAnalysis → localStorage (instant) + Firestore (background)
  → useScreenplays (React Query, 30min stale) → useFilteredScreenplays → UI
```

### Analysis Formats
- **V6** — single-pass Anthropic call with optional lenses (latam, commercial, production, coproduction)
- **V7 "Screenplay Archaeology Engine"** — 5 parallel readers (Sonnet) + synthesis roundtable; also supports Haiku triage mode

## 3. Engineering Analysis

### Strengths

**Build is clean.**
TypeScript compiles without errors. All strict checks pass. The only build warnings are expected chunk-size warnings for `vendor-react-pdf` (1.5MB gzip: 526KB) and `index.js` (627KB gzip: 192KB), which are inherent to the dependency sizes rather than organizational problems.

**Chunking strategy is well-considered.**
Manual `manualChunks` in `vite.config.ts` splits vendor code into 6 named chunks: `vendor-react`, `vendor-recharts`, `vendor-firebase`, `vendor-pdfjs`, `vendor-react-pdf`, `vendor-state`. Recharts and react-pdf are additionally lazy-loaded at the component level (AnalyticsDashboard, ComparisonModal), so they don't block initial paint.

**API keys never touch the browser.**
The Anthropic API key lives in `functions/.env`. All LLM calls route through `proxyClient.ts` → `llmProxy` Cloud Function, which calls Anthropic server-side. The original dev-only Vite proxy for `/api/anthropic` has been removed. This is correctly architected.

**Security rules are reasonable for the use case.**
`firestore.rules` gates all writes behind `request.auth != null` (anonymous auth). Reads are intentionally public (no PII in analysis data, internal tool). The rules include per-field validation (title ≤ 500 chars, text ≤ 10,000 chars) and a catch-all deny. The `analyzeScreenplay` function sanitizes titles to prevent prompt injection.

**Dual-write architecture is pragmatic.**
`analysisStore.ts` writes simultaneously to localStorage (instant, never blocks UI) and Firestore (background). On load, localStorage is returned immediately — Firestore sync is fire-and-forget. Failed Firestore writes are queued for retry. For an internal tool, this is a sensible tradeoff.

**Optimistic updates in mutations.**
`useRestoreScreenplay` implements a proper optimistic update pattern: cancel in-flight queries, update cache immediately, rollback on error. `useDeleteScreenplays` correctly invalidates both the main screenplays cache and the deleted-screenplays cache.

**Component memoization in the right places.**
`ScreenplayCard` is wrapped in `memo`. Virtual scrolling (`@tanstack/react-virtual`) is in place for the grid. These prevent the biggest render bottlenecks at 500+ items.

### Issues & Concerns

#### HIGH — Security
**`service-account.json` lives at the project root.**
It's correctly listed in `.gitignore` (line 49), so it won't be committed. But a service account JSON file at the repo root is one accidental `git add .` away from being committed. Consider moving it to `~/.config/` or using Application Default Credentials (`gcloud auth application-default login`) to remove the file from the working tree entirely.

#### MEDIUM — Build Warning
**`feedbackStore.ts` has a static/dynamic import conflict.**
Vite warns: _"feedbackStore.ts is dynamically imported by analysisService.ts but also statically imported by FeedbackSection.tsx and CalibrationPanel.tsx — dynamic import will not move module into another chunk."_ The practical effect is that feedbackStore ends up in the main bundle even though analysisService tries to load it lazily. Since the store is small this is not a performance problem now, but it signals an unresolved chunking intent. Either make all imports static or all dynamic.

#### MEDIUM — Technical Debt: Two Parallel Analysis Pipelines
Both V6 and V7 are fully active. This creates four files that must stay in sync:
- `src/lib/promptClient.ts` (V6 client prompt builder)
- `functions/src/prompts.ts` (V6 server prompt — "kept in sync manually")
- `src/lib/promptClient.v7.ts` (V7 reader prompts)
- `src/lib/multiPassAnalysis.ts` (V7 orchestrator)

The CLAUDE.md comment "kept in sync manually" on the V6 prompt is a known divergence risk. If V6 is being phased out in favor of V7, the migration path should be explicit. If both are permanent, the manual sync should be automated or at least unit-tested.

#### MEDIUM — Pre-migration Fallback Code is Dead Weight
`src/lib/api.ts` contains a full pre-migration static-JSON fetch path guarded by `isMigrationComplete()`. Per CLAUDE.md, the migration to Firestore is complete. This dead branch (~40 lines including the `migrateStaticToFirestore()` background trigger) adds cognitive load to the most-read data path. Once the migration flag is permanently `true` across all installs, this can be deleted.

#### LOW — `any` Usage in `proxyClient.ts`
Three uses of `any` in `src/lib/proxyClient.ts`:
- `const body: Record<string, any>` (line 47) — could be typed as `Record<string, string | number | Array<{role: string; content: string}>>`
- `} catch (error: any)` (line 67) — standard catch pattern, acceptable
- `let errorData: any` (line 74) — could be `unknown` with a type guard

These are in the HTTP plumbing layer and are low-risk, but they violate the project's own strict TypeScript convention.

#### LOW — Test Noise: 8 Pre-existing Failures
The test suite reports 8 failures across 2 files, all caused by Firebase Anonymous Auth making real network calls in the happy-dom environment (`auth/network-request-failed`). These are pre-existing (per project memory from 2026-03-22). They don't represent broken product logic but they create noise. The fix is to mock `signInAnonymously` in `src/test/setup.ts`.

#### LOW — normalize.ts is a 846-line Catch-all
`src/lib/normalize.ts` handles transformation from multiple historical analysis formats (V3 through V7). It's well-commented and correctly organized, but at 846 lines it's the largest single source file. As V7 fully replaces V6, consider extracting `normalizeV7Screenplay` into its own module to reduce the size.

#### LOW — README is the Vite Template README
`README.md` still contains the Vite template boilerplate (ESLint config instructions, Babel/SWC plugin notes). It doesn't describe the actual project. For an internal tool this is low-stakes but it would trip up anyone new to the codebase.

4. **Delete pre-migration fallback in `api.ts`** — once migration is confirmed complete on all installs
5. **Fix feedbackStore import conflict** — normalize all imports of feedbackStore to static
6. **Update README.md** — replace Vite template with actual project description
