## 1. What This App Is (Reconnaissance)

Internal screenplay-triage dashboard for Lemon Studios. Producers upload screenplay PDFs; the **V9 Archaeology Engine** (5 parallel Claude readers — Structure, Character, Craft, Concept, Emotion — plus a synthesis roundtable) generates coverage stored in Firestore. The dashboard provides filtering/sorting across scored dimensions, side-by-side comparison, analytics charts, PDF coverage export, DevExec AI chat (Gemini), poster generation, and shareable partner links.

**Pipeline:** browser uploads PDF → Firebase Storage `ingest-queue/` → Cloud Function trigger → VPS daemon polls, runs V9, writes `uploaded_analyses/` → browser live-syncs.

## 2. Critical Findings (fix before any team rollout)

### C1. Open LLM proxy — unauthenticated, no rate limit ⚠️ VERIFIED
`functions/src/llmProxy.ts` (`onRequest` handler): no Firebase ID-token check, no rate limiting, no per-user accounting. CORS only stops browsers — curl bypasses it entirely. Anyone who discovers the endpoint can run arbitrary Claude calls **on your Anthropic bill** (timeout is 3600s, maxInstances 50 — worst case is expensive).
**Fix:** verify Firebase ID token in the handler (`verifyIdToken`), send `Authorization: Bearer` from `src/lib/proxyClient.ts`, add per-user daily budget in Firestore, log all calls. ~half a day.

### C2. Google/Gemini API key ships to every browser ⚠️ VERIFIED
`src/stores/apiConfigStore.ts:49` bundles `VITE_GOOGLE_API_KEY`; `src/lib/analysisService.ts:295` calls `generativelanguage.googleapis.com/...?key=${googleApiKey}` directly from the client (poster generation), and `useLiveDevExec.ts` opens a Gemini Live WebSocket client-side. The key is also persisted to localStorage. Distribute this to 50 people and the key is effectively public. `VITE_TMDB_API_KEY` same story (lower risk).
Note: `.env` is **not** in git history (verified) — exposure is via the built bundle, not the repo. A previously leaked key was already rotated (commit `ce48710`).
**Fix:** route poster generation + DevExec Live through `llmProxy` (or a sibling `googleProxy` function), rotate the key, restrict it in Google Cloud console. ~1 day.

### C3. No user identity — anonymous auth only
`src/lib/firebase.ts` uses `signInAnonymously()`. All 50 users would be indistinguishable: no audit trail (who uploaded/deleted/shared what), no roles, no way to revoke one person's access, and "auth-gated writes" means *anyone who loads the app* can write/soft-delete.
**Fix:** Google Workspace sign-in (`GoogleAuthProvider` with `hd` domain restriction) + a `users/{uid}` roles collection (admin / producer / reader) + role checks in `firestore.rules`. ~2–3 days. This is the single change that most defines "team tool" vs "personal tool."

### C4. Share links never expire
`src/lib/shareService.ts` snapshots the **full analysis** (scores, verdict, weaknesses, commercial viability, optional notes + PDF URL) into `shared_views` with `allow read: if true` and no TTL. A partner link forwarded once is readable forever; revocation leaves no audit trail.
**Fix:** add `expiresAt` (default 30 days) enforced in rules, revocation log, optional view counter. ~half a day.

## 3. Scalability Blockers (the "thousands of scripts" problem)

### S1. Full-collection load, no pagination — CRITICAL at 5K
`src/lib/analysisStore.ts` / `src/lib/api.ts`: `getDocs(query(collection(...)))` fetches **every** document on load and on every cache invalidation. V9 docs run 50–150 KB each (full reader reports embedded). At 5,000 scripts that's ~500 MB per cold load and ~5,000 billed reads per user per session (~$15/day at 50 users just for viewing).
**Fix (recommended shape):** split Firestore data into a lightweight **index doc** (title, scores, genre, tier — ~2 KB) loaded in bulk, with the heavy analysis fetched **per-screenplay on modal open**. This keeps instant client-side filtering (which the UX depends on) while cutting transfer ~50×. Differential sync (`where('_savedAt', '>', lastSync)`) eliminates the invalidation cascade.

### S2. Virtualization was removed from the grid
`src/components/screenplay/ScreenplayGrid.tsx` comments say virtualization was removed because "~50–200 cards is too small to benefit." Correct then, wrong at 5,000: that's thousands of mounted cards, huge DOM, multi-second paint. `@tanstack/react-virtual` is still in dependencies — re-enable it.

### S3. localStorage mirror already at quota
`analysisStore.ts:70–103` mirrors all analyses to localStorage (~5–10 MB cap). At 500 scripts it already trips the quota-strip fallback. At 5,000 it's pure overhead. Replace with the index-doc cache or IndexedDB; drop the full-payload mirror.

### S4. Notes & favorites are browser-local — no collaboration
`notesStore.ts` and `favoritesStore.ts` persist to localStorage only. Fifty producers would each see only their own notes; nothing is shared, nothing survives a browser switch. For a triage team this defeats the purpose.
**Fix:** Firestore `users/{uid}/notes` + `users/{uid}/favorites` (depends on C3 identity), plus optionally a shared team-notes view per screenplay.

### S5. Ingest throughput — single VPS daemon
The V9 daemon processes jobs with a small worker pool. A 1,000-script backfill at ~10 min/script ≈ 55+ hours even with 3 workers. Client "budget limits" (`apiConfigStore`) are localStorage-only — not enforcement.
**Fix:** raise daemon concurrency (readers are I/O-bound API calls), make the ingest queue idempotent with retry/backoff, and move budget enforcement server-side (pairs with C1).

**Phase 0 — Now (hours): unblock the branch**
- Fix the TS6133 build error on `instrument-reskin`, commit the reskin, merge.

**Phase 1 — Security hardening (2–3 days): safe to invite anyone**
- C1 proxy auth + rate limiting + server-side budget caps
- C2 proxy Gemini calls server-side, rotate Google/TMDB keys
- C4 share-link TTL + revocation log
- Bump react-router ≥ 7.15.1, `npm audit fix`

**Phase 2 — Team identity (3–4 days): make it multi-user**
- C3 Google Workspace sign-in + roles (admin/producer/reader) + rules
- S4 notes/favorites → Firestore per-user + shared team notes
- Audit log collection (who uploaded/deleted/shared)

**Phase 3 — Scale to thousands (4–5 days)**
- S1 index-doc + lazy detail fetch + differential sync
- S2 re-enable virtual grid; S3 drop localStorage full-mirror
- S5 daemon concurrency + idempotent queue for bulk backfill

**Phase 4 — Production ops (2–3 days)**
- Sentry (frontend + functions), Firestore-sync failure surfacing
- Functions deploy in CI, staging preview channel, E2E made blocking + expanded
- Repo cleanup (binaries/assets out of git)

Total: roughly **2.5–3 weeks** of focused work to a genuinely distributable production tool.
